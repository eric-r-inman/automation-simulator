#+title:     Changelog
#+language:  en

* Upcoming

- Add the SQLite persistence layer at =crates/lib/src/db/= and
  =crates/lib/migrations/=.  One STRICT-mode init migration sets up
  every table the v0.1–v0.3 pipeline needs (property, yard, spigot,
  manifold, zone, plant, controller_instance, sensor_instance,
  sensor_reading, watering_log, schedule_item, sim_run, sim_event,
  property_design); every row carries property_id so v0.3
  multi-property storage is a data change, not a migration.  The
  =SimDb= wrapper owns a sqlx SqlitePool, runs migrations through
  the embedded =sqlx::migrate!= macro, and exposes typed
  insert/select helpers keyed to row-type structs.  Semantic
  =DbOpenError= / =MigrationError= / =QueryError= name the
  operation that failed.
- Add the deterministic simulation engine under
  =crates/lib/src/engine/=: =SimClock= + =SimInstant= + =SimDuration=
  for simulated time, =Climatology= + =WeatherModel= for monthly
  climatology with seeded stochastic rain, =soil_step= implementing
  the per-zone water-balance ODE (inflow − ET − drainage, clamped
  to [0, saturation]), =zone_inflow_mm_per_hour= converting
  catalog-level GPH into rain-equivalent flux, and =SimWorld::advance=
  stepping 1-minute sub-steps with per-zone valve state, soil state,
  and a recorded time-series history.  Ships three integration
  tests: a 30-day Portland-July reference scenario that asserts
  moisture stays in the plant-stress range plus =insta= snapshot of
  the daily series, and a =proptest= property that runs 1 000
  random inflow/ET combinations without ever producing NaN,
  negative moisture, or moisture above saturation.
- Add the hardware and species catalog at =crates/lib/src/catalog/=
  plus seed rows under =data/catalog/=.  Each category (controllers,
  sensors, weather stations, manifolds, valves, emitters, pressure
  regulators, backflow preventers, drip lines, species, soil types)
  loads from its own TOML file, cross-validates against the rest of
  the catalog, and raises semantic =CatalogLoadError= variants on
  bad input.  Growing support for new hardware in v0.3 is a data
  change only.
- Add the simulation domain model under =crates/lib/src/sim/= —
  catalog-driven property, yard, spigot, zone, manifold, plant,
  hardware-instance, and scenario types with typed-id newtypes,
  candidate =*Raw= types, semantic =thiserror= validation errors,
  and unit tests for every invariant.  Adds a compliance test that
  rejects brand names and property-specific labels inside the lib
  crate so v0.3 mix-and-match planning stays a data change, not a
  refactor.
