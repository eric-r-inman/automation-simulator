* Upcoming

* v0.1.0 (2026-04-14)

First simulator-only release.  No real-hardware drivers and no
mix-and-match planner yet (those are v0.2 and v0.3).  The full
v0.1 pipeline lands here: domain model, hardware + species
catalog, deterministic simulation engine, SQLite persistence,
property-fixture loader + =cli seed=, =Controller= /
=SensorSource= traits with simulated implementations, Axum
server with a typed sim API, Elm dashboard, and an end-to-end
demo backed by a 20-day scenario regression test.

** Engineering

- Add the Phase 9 demo + scenario regression.  =just demo= builds
  the Elm bundle and runs the server pre-loaded with the example
  property; opens at =http://127.0.0.1:3737/= for one-shot manual
  end-to-end verification.  Adds
  =data/scenarios/2026-07-heatwave.toml= — 20 simulated days
  starting July 1 with a scripted three-day heatwave and a
  daily 15-minute morning watering on Zone A1 — plus
  =data/scenarios/README.org= explaining the scenario format and
  how to add new ones.  Adds
  =crates/lib/tests/scenario_20day_demo.rs= (4 tests) that
  validate the scenario, cross-check it against the example
  property's zones, replay it through =SimWorld= and assert
  every recorded soil-moisture sample stays within
  =[0.10, 0.48]= v/v, that Zone A1 accumulates exactly 18 000
  seconds of valve open time, and that two replays produce
  byte-identical histories.
- Build the Elm dashboard (Phase 8).  Refactors =frontend/src/Main.elm=
  into a route dispatcher plus per-page modules:
  - =Page/Dashboard.elm= owns the dashboard model + update;
    fetches =/api/sim/property=, =/api/sim/state=, =/api/zones= on
    init; re-fetches state after every step / reset / run / stop.
  - =Api.elm= centralises every endpoint into one =Cmd= per call
    plus typed records and =mapN= decoders against the Phase 7a
    JSON schemas.
  - =Widget/YardMap.elm= renders an SVG yard layout (one column
    per yard, one tile per zone) with moisture-tracked fill colors
    and a contrasting stroke when a valve is open.
  - =Widget/ZoneCard.elm= shows per-zone moisture / valve / runtime
    plus Run 5 min / Run 15 min / Stop buttons.
  - =Widget/Weather.elm= shows temp + humidity + wind + solar +
    rain in one compact grid; flashes a "raining" badge on
    measurable precipitation.
  - =Widget/SimControls.elm= exposes Step (+1 hour / +6 hours /
    +1 day / +1 week) and Reset.
  Adds the =elm/svg 1.0.1= package + regenerates =elm-srcs.nix=
  / =registry.dat= for the Nix-tracked elm dependency set.
  Replaces the placeholder =public/style.css= with a hand-written
  earthy palette (deep moss-green for healthy moisture, russet
  for stress, warm stone for surfaces) plus a responsive grid that
  collapses to one column under 720 px.
- Wire the simulator into the Axum server.  =web_base::AppState=
  now also owns =SharedWorld=, =Arc<dyn Controller>=,
  =Arc<dyn SensorSource>=, =Arc<Catalog>=, and =Arc<PropertyBundle>=,
  built at startup by loading the catalog and property fixture from
  config-driven paths.  Adds five new route modules under
  =crates/server/src/routes/=:
  - =sim.rs= — =GET /api/sim/property=, =GET /api/sim/state=,
    =POST /api/sim/step=, =POST /api/sim/reset=
  - =zones.rs= — =GET /api/zones=, =POST /api/zones/{id}/run=,
    =POST /api/zones/{id}/stop= (delegates to =Controller=)
  - =sensors.rs= — =GET /api/sensors=, =GET /api/sensors/{id}/history=
    (delegates to =SensorSource=)
  - =weather.rs= — =GET /api/weather= (delegates to
    =SensorSource::weather_now=)
  Each route registers as an =aide::ApiRouter= so the schemas
  surface in the existing =/scalar= UI.  =ApiError= maps controller
  / sensor errors into the right HTTP status with a typed JSON
  body (=kind=: zone-not-found / bad-request / upstream-unreachable
  / internal-error).  Server config gains required =--property-path=
  and optional =--catalog-path= flags.
- Add the hardware-abstraction traits under =crates/lib/src/hw/=:
  =Controller= (open_zone / close_zone / list_zones / status) and
  =SensorSource= (latest_reading / history / weather_now).  Both
  are =async_trait= so =Arc<dyn ...>= works in =AppState=.  Ships
  the =SimulatedController= and =SimulatedSensorSource= impls
  backed by a shared =Arc<Mutex<SimWorld>>=; opening a zone through
  the controller side shows up in the sensor side without any
  cross-wiring.  Semantic =ControllerError= / =SensorError=
  variants name the zone and the failing operation.  v0.2 drops
  concrete real-hardware impls in alongside these without
  touching consumers.
- Tighten the dependabot ignore list: =rand_chacha >=0.9= is
  pinned alongside =rand >=0.9= since they must move in lockstep.
- Add the property-fixture loader at =crates/lib/src/seed/= and the
  =cli seed= subcommand.  Loader parses a property TOML file into
  validated domain types, cross-references it against the catalog
  and its own internal ids, and inserts the bundle into a
  migrated SQLite database.  Ships =data/properties/example-property.toml=
  (2 yards, 2 spigots, 6 zones, 10 plants, 1 controller, 6
  sensors, 1 weather station, 6 schedule items) and the
  accompanying =docs/property-schema.org= reference.  CLI is
  restructured around a =clap::Subcommand= so additional
  subcommands land without another refactor.  Amends the Phase 4
  init migration to add =weather_station_instance= alongside the
  other hardware-instance tables.
- Add the SQLite persistence layer at =crates/lib/src/db/= and
  =crates/lib/migrations/=.  One STRICT-mode init migration sets up
  every table the v0.1–v0.3 pipeline needs (property, yard, spigot,
  manifold, zone, plant, controller_instance, sensor_instance,
  sensor_reading, watering_log, schedule_item, sim_run, sim_event,
  property_design); every row carries property_id so v0.3
  multi-property storage is a data change, not a migration.  The
  =SimDb= wrapper owns a sqlx SqlitePool, runs migrations through
  the embedded =sqlx::migrate!= macro, and exposes typed
  insert/select helpers keyed to row-type structs.  Semantic
  =DbOpenError= / =MigrationError= / =QueryError= name the
  operation that failed.
- Add the deterministic simulation engine under
  =crates/lib/src/engine/=: =SimClock= + =SimInstant= + =SimDuration=
  for simulated time, =Climatology= + =WeatherModel= for monthly
  climatology with seeded stochastic rain, =soil_step= implementing
  the per-zone water-balance ODE (inflow − ET − drainage, clamped
  to [0, saturation]), =zone_inflow_mm_per_hour= converting
  catalog-level GPH into rain-equivalent flux, and =SimWorld::advance=
  stepping 1-minute sub-steps with per-zone valve state, soil state,
  and a recorded time-series history.  Ships three integration
  tests: a 30-day Portland-July reference scenario that asserts
  moisture stays in the plant-stress range plus =insta= snapshot of
  the daily series, and a =proptest= property that runs 1 000
  random inflow/ET combinations without ever producing NaN,
  negative moisture, or moisture above saturation.
- Add the hardware and species catalog at =crates/lib/src/catalog/=
  plus seed rows under =data/catalog/=.  Each category (controllers,
  sensors, weather stations, manifolds, valves, emitters, pressure
  regulators, backflow preventers, drip lines, species, soil types)
  loads from its own TOML file, cross-validates against the rest of
  the catalog, and raises semantic =CatalogLoadError= variants on
  bad input.  Growing support for new hardware in v0.3 is a data
  change only.
- Add the simulation domain model under =crates/lib/src/sim/= —
  catalog-driven property, yard, spigot, zone, manifold, plant,
  hardware-instance, and scenario types with typed-id newtypes,
  candidate =*Raw= types, semantic =thiserror= validation errors,
  and unit tests for every invariant.  Adds a compliance test that
  rejects brand names and property-specific labels inside the lib
  crate so v0.3 mix-and-match planning stays a data change, not a
  refactor.
