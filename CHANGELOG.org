* Upcoming

- Longer zone run durations on the Dashboard.  The per-zone
  card grew from two preset buttons (5 / 15 min) to five
  (5, 15, 30 min, 1 h, 2 h) plus a number-input + Run button
  for arbitrary durations.  The backend already accepts any
  positive minute count; this is purely a UI reach fix.
  Custom-minutes input state lives on the Dashboard model
  keyed by zone id so every card tracks its own.  USAGE.org
  updated to match.
- Planner BOMs now include the compute host for HAT-style
  controllers.  New =data/catalog/compute-hosts.toml= catalog
  category (four SBC rows covering Raspberry Pi 4, 5, Zero 2 W,
  and a generic 40-pin board), new =ComputeHostModel= domain
  type, and a new =requires_compute_host= flag on
  =ControllerModel=.  When the planner picks a controller with
  that flag set — the OpenSprinkler OS-Pi Pi-HAT is the first —
  it adds a =compute-host= BOM line for the cheapest
  Pi-HAT-compatible SBC in the catalog.  Controllers that ship
  with their own compute (every other row today) skip the line
  entirely.  Catalog-count snapshot updated to pin the new
  row count.  New planner test
  =hat_controllers_include_a_compute_host_line= asserts both
  directions of the invariant.  Workspace 145/145.
- Planner BOMs now actually include pipe and hose.  =compose_bom=
  gained the catalog as a parameter so it can look up real
  manifold and emitter names + prices instead of writing
  =$0.00 "(see catalog)"= placeholders, and adds two tubing
  lines: cheapest ≥ 0.4" OD drip line as mainline tubing (20 ft
  per yard + ≈ zone-area/10 ft per zone) and, when the property
  has any point-emitter zones, cheapest < 0.4" OD line as branch
  tubing (~2 ft per emitter).  BOM totals are no longer
  misleadingly low — a smoke run with a veggie bed + shrub zone
  went from $136 of components to $216.70 including tubing.
- Planner test: =bom_includes_real_prices_and_pipe_and_hose=
  asserts every line has a non-zero unit price, every expected
  category is present, and branch tubing appears whenever a
  point-emitter zone is requested.  Workspace now 144/144.
- Phase 14b multi-property registry + Properties page.
  =AppState= gains a =BTreeMap<String, PropertyBundle>=
  registry beside the currently-active bundle.  Three new
  routes: =GET /api/properties= lists everything the server
  knows about (with an =active= flag per row),
  =POST /api/properties/:id/activate= swaps the running
  simulator to the named property, and
  =DELETE /api/properties/:id= removes a dormant one from the
  registry (refuses the active property with a 400).  Each
  successful =POST /api/plan/apply= now also parks its bundle
  in the registry so the user can switch back to it later.
- Frontend: new =/properties= route wired through
  =Page/Properties.elm=.  The list highlights the active
  property, offers =Open= (activate) / =Delete= / =Dashboard →=
  actions per row, and links to the Designer for creating a new
  one.  Nav gains a "Properties" entry.
- Ergonomics: =just= with no args now runs the demo.  Added
  =just up= as a one-word shortcut, =just fmt=, and polished
  the recipe docstrings so =just --list= reads cleanly.
- Docs: new =USAGE.org= at the repo root — an end-user-facing
  tour of the Dashboard, Designer, and Properties pages, with
  the shortest path to a running demo up top.
- Tests: +6 integration tests covering the registry routes
  (initial state, apply registers + activates, activate swaps,
  unknown = 404, delete-active = 400, delete-inactive
  succeeds).  Workspace now 143/143.
- Phase 14a plan-apply bridge.  =POST /api/plan/apply= takes the
  same =PlanRequest= as =POST /api/plan= plus a =plan_index=,
  runs the recommender, and replaces the running simulator's
  property + world with the selected plan's validated
  =PropertyBundle=.  Dashboard, zone CRUD, and sensor routes all
  start serving the new property immediately.
- Designer UI gains a "Simulate this plan" button on every
  candidate card.  Clicking it calls =/api/plan/apply=, disables
  the other buttons until the apply completes, and renders a
  success / failure banner with a link to the dashboard.
- Tests: +2 server integration tests (apply swaps the running
  property; out-of-range =plan_index= returns 400).  Workspace
  total 137/137.
- Phase 13b designer UI.  New =/designer= route in the Elm SPA
  (=frontend/src/Page/Designer.elm=) drives the catalog-driven
  planner from a form: property metadata, budget, smart-preference
  and pressure-comp toggles, and a nested list of yards each with
  their own list of zones (plant kind, name suffix, area).  "+ Add
  yard" and "+ Add zone" buttons expand the form; "Remove" buttons
  on each row keep iteration cheap.  Submit POSTs =/api/plan= and
  renders up to N candidate plans side by side, each with its
  controller, score, BOM total, rationale bullets, and a
  collapsible full bill of materials.  Clear error text whenever
  the form or the server rejects the request.
- =Api.elm= gains =PlanRequest=, =PlanResponse=, =Plan=, =PlanBom=,
  =BomLine=, plus =postPlan= and the matching encoders/decoders.
- Top-level nav gains a "Designer" link; =Main.elm= routes
  =/designer= to the new page.
- Phase 13a catalog-driven planner (backend).  New
  =crates/lib/src/planner/= module: =PropertyRequirements= plus
  a deterministic =planner::recommend= that enumerates up to
  three controller options (cheapest, smart-preferred,
  most-capable), one manifold per yard, emitter per plant kind,
  a 25-PSI regulator, and a backflow preventer, scores the
  results, and returns ranked =PropertyPlan=s.  Each plan
  carries a validated =PropertyBundle=, a bill of materials,
  and a short rationale.  The planner is strictly
  catalog-driven: brand names live only in TOML, never in
  source.  Added an =is_smart= field on =ControllerModel= so
  the smart-preference heuristic is a catalog fact, not a
  hard-coded string list.
- =POST /api/plan= wraps the recommender over HTTP.  DTO layer
  accepts kebab-case plant kinds, clamps =top_n= to [1, 10],
  and maps unsatisfiable-requirement variants of
  =PlannerError= to 400 (semantic user errors) versus 500
  (internal invariants).  Scalar docs pick up the request /
  response schemas automatically.
- Tests: +6 planner unit tests
  (=crates/lib/tests/planner_recommend.rs=: non-empty plans,
  sorted-by-score, determinism, zero-zones and unknown-soil
  errors, smart-preference resolution) and +3 server
  integration tests for =/api/plan= (ranked candidates, bad
  plant kind → 400, zero zones → 400).  Full workspace now at
  135/135 passing.
- Phase 12 catalog breadth.  Grow =data/catalog/= so the
  dropdowns and the v0.3 planner have real choices to pick
  from.  +7 controllers (now 8: OpenSprinkler OS-Pi, Rachio 3
  8/16-zone, Hunter Pro-C, Orbit B-hyve, Rain Bird ESP-Me,
  generic 24 VAC, plus the original example).  +5 sensors (now
  7: Ecowitt GW2000 gateway, Ecowitt WH51 soil, Davis Vantage
  Vue soil, Rachio Wireless Flow Meter, generic resistive).
  +3 weather stations (now 4: Ecowitt WS90, Davis Vantage Vue,
  Ambient WS-2902).  +3 manifolds (now 4: Rain Bird HV300,
  Orbit 3-zone, Hunter ICV).  +4 emitters (now 9: Netafim
  Techline CV in 12"/18" spacing, Dramm 1 GPH PC, Rain Bird
  Xeri-Bird).  +3 pressure regulators (now 4).  +2 backflow
  preventers (now 3).
- Add the controller-capacity cross-validator at property load:
  =SeedError::ControllerOverCapacity= fires when a controller
  is wired to more zones than its catalog model allows.  Catches
  a real planning mistake before the simulator starts.
- Add three insta snapshot tests
  (=crates/lib/tests/catalog_snapshot.rs=) so the seeded catalog
  row counts and the controller / emitter id lists are pinned;
  PRs that grow the catalog must accept the snapshot under
  review.
- Wire zone CRUD into the dashboard.  Each zone card gains
  =Edit= and =Delete= buttons; a =+ Add zone= button at the top
  of the Zones section opens a modal form with dropdowns for
  yard, manifold, plant kind, emitter spec, and soil type
  (populated from the loaded property and a new
  =GET /api/catalog= endpoint).  Successful submissions refetch
  the dashboard so the cards reflect the new state immediately.
  Delete prompts a confirmation dialog before firing.
- Backend additions to support the editor:
  =GET /api/sim/property= now also returns =manifolds= so the
  editor's manifold dropdown has data.  A new =GET /api/catalog=
  endpoint returns the emitter, soil-type, and species rows the
  form needs.  Frontend Api.elm gains the matching types +
  =fetchCatalog= / =postCreateZone= / =postUpdateZone= /
  =deleteZone= helpers.
- New widget =frontend/src/Widget/ZoneEditor.elm= drives the
  modal form; styling lives in =public/style.css= as a hand-
  written modal backdrop + grid-based form rows that collapse
  to one column under 540 px.
- Add zone-definition CRUD endpoints.  =POST /api/zones/definitions=
  creates a zone, =PATCH /api/zones/definitions/{id}= edits one,
  =DELETE /api/zones/definitions/{id}= removes one.  All three
  validate against the property's existing yards/manifolds and the
  hardware/species catalog before mutating; both the in-memory
  =PropertyBundle= and the running =SimWorld= are updated
  atomically (under their respective locks).  v0.1 keeps these
  changes in memory only — restart reloads the property TOML.
- Add =SimWorld::add_zone= / =update_zone= / =remove_zone= with a
  new =SimWorldError::DuplicateZone= variant; soil + valve state
  + recorded history are kept consistent on each mutation.
- =AppState.property= moves from =Arc<PropertyBundle>= to
  =Arc<tokio::sync::Mutex<PropertyBundle>>= so the CRUD routes
  can mutate the property roster without restarting.
- Drop the dashboard's =Yard map= section.  The per-zone cards
  carry the same data without the SVG layout overhead, and the
  layout was a fixture-keyed approximation rather than the
  property's actual geometry.  Removes =Widget/YardMap.elm=, the
  corresponding =dashboard-map= section in =Page/Dashboard.elm=,
  and the =elm/svg 1.0.1= dependency from =elm.json= +
  =elm-srcs.nix= + =registry.dat=.

* v0.1.0 (2026-04-14)

First simulator-only release.  No real-hardware drivers and no
mix-and-match planner yet (those are v0.2 and v0.3).  The full
v0.1 pipeline lands here: domain model, hardware + species
catalog, deterministic simulation engine, SQLite persistence,
property-fixture loader + =cli seed=, =Controller= /
=SensorSource= traits with simulated implementations, Axum
server with a typed sim API, Elm dashboard, and an end-to-end
demo backed by a 20-day scenario regression test.

** Engineering

- Add the Phase 9 demo + scenario regression.  =just demo= builds
  the Elm bundle and runs the server pre-loaded with the example
  property; opens at =http://127.0.0.1:3737/= for one-shot manual
  end-to-end verification.  Adds
  =data/scenarios/2026-07-heatwave.toml= — 20 simulated days
  starting July 1 with a scripted three-day heatwave and a
  daily 15-minute morning watering on Zone A1 — plus
  =data/scenarios/README.org= explaining the scenario format and
  how to add new ones.  Adds
  =crates/lib/tests/scenario_20day_demo.rs= (4 tests) that
  validate the scenario, cross-check it against the example
  property's zones, replay it through =SimWorld= and assert
  every recorded soil-moisture sample stays within
  =[0.10, 0.48]= v/v, that Zone A1 accumulates exactly 18 000
  seconds of valve open time, and that two replays produce
  byte-identical histories.
- Build the Elm dashboard (Phase 8).  Refactors =frontend/src/Main.elm=
  into a route dispatcher plus per-page modules:
  - =Page/Dashboard.elm= owns the dashboard model + update;
    fetches =/api/sim/property=, =/api/sim/state=, =/api/zones= on
    init; re-fetches state after every step / reset / run / stop.
  - =Api.elm= centralises every endpoint into one =Cmd= per call
    plus typed records and =mapN= decoders against the Phase 7a
    JSON schemas.
  - =Widget/YardMap.elm= renders an SVG yard layout (one column
    per yard, one tile per zone) with moisture-tracked fill colors
    and a contrasting stroke when a valve is open.
  - =Widget/ZoneCard.elm= shows per-zone moisture / valve / runtime
    plus Run 5 min / Run 15 min / Stop buttons.
  - =Widget/Weather.elm= shows temp + humidity + wind + solar +
    rain in one compact grid; flashes a "raining" badge on
    measurable precipitation.
  - =Widget/SimControls.elm= exposes Step (+1 hour / +6 hours /
    +1 day / +1 week) and Reset.
  Adds the =elm/svg 1.0.1= package + regenerates =elm-srcs.nix=
  / =registry.dat= for the Nix-tracked elm dependency set.
  Replaces the placeholder =public/style.css= with a hand-written
  earthy palette (deep moss-green for healthy moisture, russet
  for stress, warm stone for surfaces) plus a responsive grid that
  collapses to one column under 720 px.
- Wire the simulator into the Axum server.  =web_base::AppState=
  now also owns =SharedWorld=, =Arc<dyn Controller>=,
  =Arc<dyn SensorSource>=, =Arc<Catalog>=, and =Arc<PropertyBundle>=,
  built at startup by loading the catalog and property fixture from
  config-driven paths.  Adds five new route modules under
  =crates/server/src/routes/=:
  - =sim.rs= — =GET /api/sim/property=, =GET /api/sim/state=,
    =POST /api/sim/step=, =POST /api/sim/reset=
  - =zones.rs= — =GET /api/zones=, =POST /api/zones/{id}/run=,
    =POST /api/zones/{id}/stop= (delegates to =Controller=)
  - =sensors.rs= — =GET /api/sensors=, =GET /api/sensors/{id}/history=
    (delegates to =SensorSource=)
  - =weather.rs= — =GET /api/weather= (delegates to
    =SensorSource::weather_now=)
  Each route registers as an =aide::ApiRouter= so the schemas
  surface in the existing =/scalar= UI.  =ApiError= maps controller
  / sensor errors into the right HTTP status with a typed JSON
  body (=kind=: zone-not-found / bad-request / upstream-unreachable
  / internal-error).  Server config gains required =--property-path=
  and optional =--catalog-path= flags.
- Add the hardware-abstraction traits under =crates/lib/src/hw/=:
  =Controller= (open_zone / close_zone / list_zones / status) and
  =SensorSource= (latest_reading / history / weather_now).  Both
  are =async_trait= so =Arc<dyn ...>= works in =AppState=.  Ships
  the =SimulatedController= and =SimulatedSensorSource= impls
  backed by a shared =Arc<Mutex<SimWorld>>=; opening a zone through
  the controller side shows up in the sensor side without any
  cross-wiring.  Semantic =ControllerError= / =SensorError=
  variants name the zone and the failing operation.  v0.2 drops
  concrete real-hardware impls in alongside these without
  touching consumers.
- Tighten the dependabot ignore list: =rand_chacha >=0.9= is
  pinned alongside =rand >=0.9= since they must move in lockstep.
- Add the property-fixture loader at =crates/lib/src/seed/= and the
  =cli seed= subcommand.  Loader parses a property TOML file into
  validated domain types, cross-references it against the catalog
  and its own internal ids, and inserts the bundle into a
  migrated SQLite database.  Ships =data/properties/example-property.toml=
  (2 yards, 2 spigots, 6 zones, 10 plants, 1 controller, 6
  sensors, 1 weather station, 6 schedule items) and the
  accompanying =docs/property-schema.org= reference.  CLI is
  restructured around a =clap::Subcommand= so additional
  subcommands land without another refactor.  Amends the Phase 4
  init migration to add =weather_station_instance= alongside the
  other hardware-instance tables.
- Add the SQLite persistence layer at =crates/lib/src/db/= and
  =crates/lib/migrations/=.  One STRICT-mode init migration sets up
  every table the v0.1–v0.3 pipeline needs (property, yard, spigot,
  manifold, zone, plant, controller_instance, sensor_instance,
  sensor_reading, watering_log, schedule_item, sim_run, sim_event,
  property_design); every row carries property_id so v0.3
  multi-property storage is a data change, not a migration.  The
  =SimDb= wrapper owns a sqlx SqlitePool, runs migrations through
  the embedded =sqlx::migrate!= macro, and exposes typed
  insert/select helpers keyed to row-type structs.  Semantic
  =DbOpenError= / =MigrationError= / =QueryError= name the
  operation that failed.
- Add the deterministic simulation engine under
  =crates/lib/src/engine/=: =SimClock= + =SimInstant= + =SimDuration=
  for simulated time, =Climatology= + =WeatherModel= for monthly
  climatology with seeded stochastic rain, =soil_step= implementing
  the per-zone water-balance ODE (inflow − ET − drainage, clamped
  to [0, saturation]), =zone_inflow_mm_per_hour= converting
  catalog-level GPH into rain-equivalent flux, and =SimWorld::advance=
  stepping 1-minute sub-steps with per-zone valve state, soil state,
  and a recorded time-series history.  Ships three integration
  tests: a 30-day Portland-July reference scenario that asserts
  moisture stays in the plant-stress range plus =insta= snapshot of
  the daily series, and a =proptest= property that runs 1 000
  random inflow/ET combinations without ever producing NaN,
  negative moisture, or moisture above saturation.
- Add the hardware and species catalog at =crates/lib/src/catalog/=
  plus seed rows under =data/catalog/=.  Each category (controllers,
  sensors, weather stations, manifolds, valves, emitters, pressure
  regulators, backflow preventers, drip lines, species, soil types)
  loads from its own TOML file, cross-validates against the rest of
  the catalog, and raises semantic =CatalogLoadError= variants on
  bad input.  Growing support for new hardware in v0.3 is a data
  change only.
- Add the simulation domain model under =crates/lib/src/sim/= —
  catalog-driven property, yard, spigot, zone, manifold, plant,
  hardware-instance, and scenario types with typed-id newtypes,
  candidate =*Raw= types, semantic =thiserror= validation errors,
  and unit tests for every invariant.  Adds a compliance test that
  rejects brand names and property-specific labels inside the lib
  crate so v0.3 mix-and-match planning stays a data change, not a
  refactor.
