* Context and goal

We are building a simulator for an automatic, multi-zone food forest
irrigation system inside the rust-template workspace at
[[file:./][./]].  The work proceeds in three stages:

1. *Simulator-first (v0.1)* — a fully working simulation of the real
   property at *the example property (a residential lot in Portland, Oregon)* — 6
   irrigation zones across two spigots (front + back), no physical
   hardware required.  The whole downstream pipeline (schedules, UI,
   history, recommendations) runs against this simulation.
2. *Real hardware (v0.2)* — once the simulator is convincing, we add
   drivers that talk to actual OpenSprinkler and Ecowitt hardware
   behind the same traits.  The homeowner at the example property is the
   first real-world deployment.
3. *Mix-and-match planner (v0.3)* — once the simulator is working for
   the test-case real property, we add capacity to plan for any real
   property with mix-and-match tech (hardware and software) setup.
   This means: arbitrary property geometry and plantings, a catalog of
   supported controllers / sensors / valves / emitters, and a
   recommendation engine that turns a property description into a
   costed, compatible bill of materials that the user can then
   simulate against.

The single-trait-surface decision from stage 1 is what makes stages 2
and 3 cheap: every concrete controller / sensor / weather source is
just another implementation of the traits the rest of the system is
already written against.

Success for v0.1 is: a user loads the seed property, advances
simulated time, and sees zones drip, soil moisture rise and fall, and
scheduled runs fire — all in the browser, served by the Axum server
in this workspace.  Stages 2 and 3 have their own success criteria
captured in Phases 11–14 below.

* Scope boundary across the three stages

| Concern                  | Simulator (v0.1)                                                        | Real hardware (v0.2)                             | Planner / mix-and-match (v0.3)                                                                   |
|--------------------------+-------------------------------------------------------------------------+--------------------------------------------------+--------------------------------------------------------------------------------------------------|
| Property definition      | Single committed TOML fixture in =data/properties/=                     | Same fixture + live calibration overrides        | User-authored via UI; stored in SQLite; exportable as TOML                                       |
| Hardware choice          | Hardcoded to the the example setup (OpenSprinkler OS-Pi, Ecowitt, etc.)        | Same as v0.1                                     | Selected from a catalog of supported hardware; recommendation engine proposes an initial picklist |
| Controller               | =SimulatedController= impl of the =Controller= trait                    | =OpenSprinklerController= speaking HTTP to OS-Pi | Any compatible =Controller= impl; the catalog declares which are supported                       |
| Sensors                  | =SimulatedSensorSource= derived from the physics model                  | =EcowittSensorSource= receiving HTTP pushes      | Any compatible =SensorSource= impl; planner scores properties on sensor coverage                 |
| Weather                  | Portland climatology + seeded stochastic rain events                    | =EcowittWeatherSource= (WS90 readings)           | User picks climatology by lat/long; real station optional                                        |
| Soil moisture            | ODE: irrigation inflow − ET − drainage                                  | Measured, fed through the same downstream pipeline | Per-zone soil parameters from the property spec; simulator still validates the plan              |
| Time                     | =SimClock= advanced in discrete steps, deterministic                    | Wall-clock via =chrono::Utc::now=                | Mixed — planner runs simulator over a proposed design before committing                          |
| Wiring between stages    | Runtime config flag =hardware_mode= = =simulated= \vert =real= \vert =planning= | Same flag set to =real=                     | Same flag set to =planning=                                                                      |

Every piece of application logic — schedules, alerting, UI, history,
recommendations — is written against the traits, not against either
driver.  The switch between simulation, reality, and planning is a
single config value plus (for planning) a property picker in the UI.

* Template compliance requirements we must honor

These are imposed by the rust-template scaffold and [[file:CLAUDE.md][CLAUDE.md]].  Every
phase below assumes them.

- Workspace layout: add modules under =crates/lib=, =crates/cli=,
  =crates/server=; do not create new top-level crates without need.
- =main.rs= stays an orchestrator; delegate to sibling modules.
- Staged configuration: =CliRaw → ConfigFileRaw → Config=.  New
  knobs land in all three.
- Errors: semantic variants via =thiserror=, named for /what/ failed
  (=ScenarioLoadError=, =ZoneOpenError=), never generic.
- No =anyhow= wrappers that strip context.  No =Option= for fields
  that are conceptually required — use candidate types and validate.
- Comments: complete sentences, 80 columns, non-obvious intent only.
- Pascal initialisms (=Url=, =Api=, =Os=), not preserved (=URL=, =API=).
- Prefer combinators and =tap= over temporary locals; avoid early
  =return= where a chain reads better.
- All docs are =.org= (no =.md= unless a tool demands it).
- All =cargo= commands run inside =nix develop=.  Pre-commit hook in
  =.cargo-husky= enforces =treefmt= + =cargo test --workspace=.
- Server infrastructure (healthz, metrics, /me, /scalar, OIDC, systemd
  socket activation, watchdog) is already provided by the template —
  reuse it rather than duplicating.
- License: AGPL-3.0-or-later; lib crate is publish = true (public
  repo, crates.io workflow emitted).
- Frontend is Elm, compiled by =elm make= into =frontend/public/elm.js=.
  No JS framework, no Tailwind.  Styling is hand-written CSS.

* Phase 2 — Domain model in =crates/lib/src/sim/=

Goal: data types that describe a property, its zones, plants, and
physical hardware, as pure Rust with no I/O.  Unit-tested invariants.
/Designed to be catalog-driven from the start/ so v0.3 mix-and-match
planning is a data-and-UI exercise, not a refactor.

** Steps

1. =property.rs=: =Property= (address, lot-area, spigots, yards,
   climate-zone); =Yard= (name, square-footage); =Spigot= (id,
   position, mains-pressure-psi).  Yards are a named collection, not a
   =Front= / =Back= enum — arbitrary properties may have side yards,
   courtyards, greenhouses.  Implement =TryFrom<PropertyRaw>= so the
   TOML loader can produce validated =Property= from untrusted input.
2. =zone.rs=: =Zone= (id, yard-id, plant-kind, emitter-spec-id,
   area-sq-ft, soil-type-id, manifold-id).  =plant-kind= is an enum
   that starts with =VeggieBed= / =Shrub= / =Perennial= / =Tree=
   (=Tree= explicitly /not irrigated/ for the example but supported for other
   properties).  =emitter-spec-id= and =soil-type-id= reference the
   catalog so the zone carries an id, not inlined datasheet values.
   Invariants: every zone belongs to exactly one manifold which
   belongs to exactly one spigot; =emitter-spec= matches =plant-kind=
   recommendations (soft-warned, not hard-errored).
3. =plant.rs=: =Plant= (id, zone-id, species-id, planted-on,
   water-needs-ml-per-day, notes).  =species-id= references the
   catalog.
4. =hardware.rs=: /instance/ types that reference the catalog — each
   is an id plus placement/wiring data, not a datasheet.  Examples:
   =ControllerInstance= (id, model-id, position, assigned-zones),
   =ManifoldInstance= (id, model-id, spigot-id, zone-ids),
   =SensorInstance= (id, model-id, zone-id, gateway-id), etc.  The
   datasheet-level constants live in the catalog (Phase 2.5); a
   property owns instances, never model definitions.
5. =scenarios.rs=: =Scenario= (seed, start-date, duration, weather
   overrides, manual interventions).  Deterministic with the RNG seed.
6. Unit tests per file exercising at least one invariant violation
   (which must produce a semantic error) and one happy path.
7. Explicitly test that nothing in the domain model has hard-coded
   references to "front yard", "back yard", OpenSprinkler-specific
   terms, or the six zones from the example.  Those names live in the
   fixture, not the model.

** Verification

- =cargo test -p automation-simulator-lib sim::= passes.
- =cargo doc -p automation-simulator-lib --no-deps= produces clean
  docs (no warnings).
- =rg -i "front yard\|back yard\|opensprinkler\|ecowitt" crates/lib/src/=

* Phase 2.5 — Hardware and plant catalog

Goal: a data-driven catalog of supported hardware models and plant
species that the domain model references by id.  For v0.1 we seed
only the models present at the example property, but the catalog
shape is final so v0.3 adds rows, not columns.

** Steps

1. =crates/lib/src/catalog/= module with one submodule per category:
   =controllers.rs=, =sensors.rs=, =weather_stations.rs=,
   =valves.rs=, =manifolds.rs=, =emitters.rs=, =pressure_regulators.rs=,
   =backflow_preventers.rs=, =drip_lines.rs=, =spray_heads.rs=, =pumps.rs=,
   =species.rs=, =soil_types.rs=.
2. Each model struct includes id (stable string), human name,
   manufacturer, price estimate (USD), physical constants, and a
   =compatibility= field declaring what it can pair with (e.g. the
   OS-Pi HAT declares =requires: RaspberryPi=; the WH51 declares
   =gateway: EcowittGw2000=).
3. =data/catalog/= directory with one TOML per category containing the
   seeded rows.  v0.1 seeds exactly the the example setup: OS-Pi controller,
   GW2000 gateway, 6× WH51 sensors, WS90 weather station, Rain Bird
   HV300 manifolds, Orbit backflow, 25 PSI regulator, Y-filter,
   1.0 GPH pressure-compensating emitters, 12"-spaced inline drip,
   micro-spray heads, etc.  Every line item from the docx Section 2
   shopping list becomes a catalog row.
4. =crates/lib/src/catalog/mod.rs=: =Catalog::load(dir)= →
   =Result<Catalog, CatalogLoadError>= that parses every TOML and
   cross-validates compatibility references (no broken ids).
5. =ScenarioWorld= / =SimWorld= takes =&Catalog= when it needs
   datasheet constants; nothing outside the catalog hardcodes them.
6. Unit tests: a malformed compatibility reference fails loudly; a
   valid catalog loads; the the example fixture resolves every id it names.

** Verification

- =cargo test -p automation-simulator-lib catalog::= passes.
- Loading =data/catalog/= in an integration test produces a non-empty
  =Catalog= with every id the the example fixture references.
- Adding a synthetic =controllers/rachio-3.toml= row does /not/ break
  any tests — the catalog is additive.

* Phase 3 — Simulation engine core

Goal: a deterministic =tick= function that evolves the world forward
by a configurable number of simulated minutes, and the =SimClock=
that drives it.

** Steps

1. =clock.rs=: =SimClock= (start-instant, now, step).  All code below
   takes =&SimClock= or =SimInstant= rather than =chrono::Utc::now()=.
2. =weather.rs=: =WeatherModel= producing temperature, humidity, wind,
   solar radiation, precipitation for a given =SimInstant=.  Seeded
   Portland climatology: monthly normals from NOAA, with stochastic
   rain events sampled from a seeded =ChaCha8Rng=.
3. =soil.rs=: per-zone soil-moisture ODE.  =d(theta)/dt = I − ET − D=
   where =I= is irrigation inflow (computed from valve state and
   emitter spec), =ET= is Penman-Monteith evapotranspiration (pulled
   from =WeatherModel=), =D= is deep drainage (saturated flow past
   field capacity).  State: volumetric water content per zone, clamped
   to [0, saturation].
4. =flow.rs=: when valves are open, compute inflow rate from emitter
   count × gph × pressure-compensation.  Handle pressure drop across
   manifold and drip line runs.
5. =tick.rs=: top-level =SimWorld::advance(duration)= that steps the
   clock in 1-minute sub-steps, calling weather → flow → soil →
   recording sensor samples.
6. Integration test: load a tiny property (1 zone, 1 plant), run a
   30-day scenario with a fixed schedule, assert soil moisture stays
   within [0.15, 0.40] v/v the whole time.  Snapshot the time series
   with =insta=.
7. Property test: over 1,000 random schedules, no step ever produces
   NaN, negative moisture, or moisture above saturation.

** Verification

- =cargo nextest run -p automation-simulator-lib= passes.
- The =insta= snapshot for the reference scenario is stable across
  three consecutive runs (determinism check).

* Phase 4 — Persistence layer

Goal: a SQLite schema that holds property definitions, sensor readings,
watering log, plant inventory, and simulation-run history.  Readable
by both =cli= and =server=.

** Steps

1. Generate migrations with =sqlx migrate add -r init_schema= inside
   the dev shell.  Tables: =property=, =yard=, =spigot=, =manifold=,
   =zone=, =plant=, =controller_instance=, =sensor_instance=,
   =sensor_reading=, =watering_log=, =sim_run=, =sim_event=,
   =schedule_item=, =property_design= (v0.3 draft state).  The schema
   supports multiple properties per database from day one (one row in
   =property= with a stable id; everything else carries
   =property_id=) even though v0.1 only ever has one row.
2. =crates/lib/src/db/models.rs=: =sqlx::FromRow= types mirroring the
   tables.  One file per table, one struct per file.
3. =crates/lib/src/db/mod.rs=: =Pool= wrapper =SimDb= with
   =SimDb::connect(path) -> Result<Self, DbOpenError>= and
   =SimDb::migrate(&self) -> Result<(), MigrationError>=.
4. Add =sqlx-data.json= generation to the pre-commit flow (=cargo
   sqlx prepare --workspace=) so offline builds work.
5. Integration test: open a =:memory:= DB, run migrations, insert
   one of each record, read it back.

* Phase 13 (v0.3) — Property designer and BOM planner

Goal: the user inputs property dimensions, garden layout, and
preferences, and the app returns a costed bill of materials plus a
runnable simulation of the proposed design.

** Steps

1. =crates/lib/src/planner/= module.
   - =requirements.rs=: =PropertyRequirements= (yards, planting
     intents per area, watering budget, smart-vs-dumb preference,
     budget cap, compatibility overrides).
   - =plan.rs=: =PropertyPlan= = =Property= + =Bom= + =Rationale=.
     Derived deterministically from =PropertyRequirements= + =Catalog=.
   - =scoring.rs=: scoring rules for candidate plans — coverage, cost,
     smart-features, future-expandability, simplicity.  Pure functions
     exposed for tests.
   - =recommend.rs=: =recommend(requirements, catalog) -> Vec<PropertyPlan>=

   requirements, response: top candidates); =POST /api/plan/{id}/simulate=

   v0.1 scenario against it); =POST /api/plan/{id}/commit= (turns
   the candidate into a persisted =property_design= row the user can
   iterate on).
3. =crates/cli/src/commands/plan.rs=: headless CLI that accepts a
   requirements TOML and prints the top-N plans as TOML — useful for
   CI and for power users.
4. Deterministic tests: fixed requirements → fixed top plan across
   runs.  =insta= snapshots guard the output.

* Risks and mitigations

| Risk                                               | Mitigation                                                                                                                   |
|----------------------------------------------------+------------------------------------------------------------------------------------------------------------------------------|
| Soil-moisture physics is subtly wrong              | Snapshot tests + explicit bounds assertions per scenario; document all constants; bias toward over-watering in uncertainty. |
| Elm HTTP wiring gets tedious without codegen       | Accept the boilerplate for v0.1; reconsider generators only if we add ≥3 more resource types after launch.                  |
| sqlx offline prepare drifts                        | Pre-commit hook runs =cargo sqlx prepare --check=; CI fails loudly on drift.                                                 |
| Changing trait shapes late breaks both drivers     | Design Phase 6 traits with the real drivers in mind from day one; review the trait file against the docx Section 7.3.       |
| Scope creep into the real-hardware path            | Phase 11 is explicitly deferred.  Do not start it until Phase 10 is tagged.                                                  |
| Elm/Rust type duplication for API payloads         | Single source of truth lives in =crates/lib/src/api_types.rs=; Elm decoders hand-written once, reviewed against openapi.json. |
| Catalog grows faster than tests can cover          | Phase 12 catalog additions must come with at least one compatibility test; CI fails on an unchecked row.                     |
| Planner produces nonsense recommendations          | Scorer rules live in pure functions with =insta= snapshots on fixed inputs; catalog rules (Phase 12) reject hard errors.     |
| v0.3 designer UI drifts from v0.1 dashboard        | Both consume the same =Api.elm= module; the /Simulate this plan/ button reuses the v0.1 dashboard pointed at a different id. |

* Open questions to resolve during Phase 2

These are assumptions we are making now and will confirm or correct
once we have the domain model in hand.

1. How many plants per zone are worth seeding?  Default assumption:
   3–5 representative plants per zone, not a full inventory — the
   homeowner fills the rest in through the UI.
2. Which soil type should the fixture use?  Default assumption: silty
   clay loam, the most common Portland residential soil per Oregon
   State Extension, applied uniformly; zone-level overrides allowed.
3. What are the mains-pressure numbers at the example property?
   Default assumption: 60 psi, which is the Portland Water Bureau
   typical; the pressure regulators downstream normalize to 25 psi so
   the exact mains figure has minor effect on the simulation.
4. Does the homeowner want OIDC auth on v0.1?  Default assumption:
   no — v0.1 runs with =base_url= set and no OIDC configured.  The
   template's OIDC scaffolding stays in place for the real
   deployment.

* Out of scope for v0.1

- Mobile native apps.  Browser only.
- Alerting (email, SMS, push).
- Real-hardware drivers — deferred to v0.2 (Phase 11).
- Multi-property UI and mix-and-match planning — deferred to v0.3
  (Phases 12–14).  The schema and catalog already support it; v0.1
  simply does not expose the corresponding UI or API routes.
- Machine-learning-driven scheduling.  We stick to deterministic
  rules.
- Historical weather import from NOAA live API.  Use built-in
  climatology only.
